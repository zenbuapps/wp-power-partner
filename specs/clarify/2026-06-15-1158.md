# Clarify Session 2026-06-15 11:58

## Idea

為 Power Partner WordPress 外掛「把所有的業務流程情境的整合測試案例都補上」。依 AIBDD 訪談流程釐清「所有業務流程情境」的精確範圍與驗收標準，產出結構化規格（.feature / activities / execution plan），最後交接下游 agent（預期是 test-creator 實作）。

三大已識別缺口：
1. 開站 SiteSync 本體無 IT — Product/SiteSync.php 的 site_sync_powercloud() / site_sync_by_subscription()
2. 授權碼 LC 生命週期無 IT — Domains/LC/Core/LifeCycle.php 的建立/停用/恢復
3. DisableHooks 排程編排薄 — Domains/Site/Core/DisableHooks.php 的排程/取消/重啟編排層

## Q&A

- Q: 「所有業務流程情境」的邊界？ → A: A — 聚焦三大真實缺口（SiteSync 開站本體 / LC 全生命週期 / DisableHooks 排程編排層），補 happy + edge + failure；既有已覆蓋的不重造。
- Q: 驗收門檻？ → A: A — 以「情境清單逐條對應 test method + 全綠」驗收；不裝 coverage driver、不動 CI。
- Q: 雙後端對等程度？ → A: A — PowerCloud 等量覆蓋；WPCD 只補關鍵差異路徑（停用/啟用/開站分支 routing 與備註），不為 WPCD 補 Email/回調全套。
- Q: 測試層級分工？ → A: A — 內部編排邏輯（hook 觸發、meta 寫入、ActionScheduler 排程、as_schedule_single_action 參數、備註內容）放 PHP IT；E2E 不動。
- Q: send_email 延遲發信（4 分鐘）斷言策略？ → A: A — 斷言 as_next_scheduled_action（hook + args + 時間窗）+ 直接呼叫 SiteSync::send_email() 驗 email_payloads_tmp 讀取→發信→刪 meta 三段。

## 校正後的事實（訪談中經 grep 驗證）

- 缺口 #1 SiteSync 開站本體：site_sync_by_subscription 在 IT 中僅出現於 ProductMetaTest 的一行註解，零實際呼叫；site_sync_powercloud / send_email 全無 → 確認 0 IT。
- 缺口 #2 LC LifeCycle：LifeCycle / create_lcs / subscription_failed / subscription_success 在 IT 中零引用 → 確認 0 IT。
- 缺口 #3 DisableHooks：SiteDisableRoutingTest（issue #18, 10 tests）已覆蓋 DisableSiteScheduler::action_callback（被執行的關站 callback + host_type 路由）。真正真空是 schedule_disable_site（N 天 timestamp）、cancel_disable_site_schedule、restart_all_stopped_sites_scheduler（啟用路徑 + issue #18 啟用失敗備註）→ 這三個方法 IT 零引用。
